# 服务器稳定性深度优化实施方案

## 背景与结论

本方案覆盖常规赛（`src/main.opy`）和 ARAM（`src/aramMain.opy`），目标是降低长期服务器计算压力、消除可避免的高频重复查询，并保持现有玩法效果与英雄目标语义。

- 不使用 `getAverageServerLoad()` 或 Anti Crash 作为运行时保护。该采样本身持续增加服务器负担，已不符合本项目的实测策略。
- 本文的“特效”指技能特殊效果：附加伤害、状态、生命池、DOT/HOT、位移、技能刷新、持续属性变化等；视觉对象不是优化主体。
- Workshop 子程序按顺序执行。优化重点是减少单条执行链的重复工作、限制状态规模和控制触发频率，而非为并发子程序增加锁。
- 优化以固定节流、玩家槽位错峰、硬上限和事件内结果复用替代运行时负载采样。

## 当前基线

截至本方案建立时，源码中有 51 处有效 `getAverageServerLoad()` 调用。它们分布在 AI 控制、移动、生成/生命周期、ARAM 初始化和少数英雄规则中，主要用于动态计算 `wait(...)` 时长。

`pnpm run perf:scan:strict` 当前无 HIGH 或 MEDIUM 阻断项，但报告 93 个 LOW 热点。热点以以下模式为主：

- 高频规则中的 `getPlayersInRadius(...)`、`getLivingPlayers(...)`、`isInLoS(...)` 与 `distance(...)`。
- 对同一批玩家重复 `filter`、列表推导与 `sorted(...)`。
- 目标选择先做半径检测，再重建存活玩家列表、LoS 筛选、状态过滤和排序。

共享 AI 的 `[AI]: Target in line of sight` 是优先路径：它在每个符合条件的 AI 循环中依次执行半径查询、全体存活敌人 LoS 扫描、不可用目标过滤和按 `health × distance` 排序。其刷新间隔虽然为 0.512 秒，但同一轮内仍重复构造候选集合。

## 不可改变的行为边界

- 不改变英雄技能数值、半径、持续时间、冷却、目标优先级公式或原生状态效果。
- 不将当前按英雄拆分、带 `defineMember` 命名的小型状态数组迁移为大型魔法下标数组。
- 不新增持续性的服务器负载监控规则。
- 不触碰入口 include 顺序、公共变量/子程序声明顺序或 Hero init/reset 合同，除非某阶段明确完成相应合同验证。
- 不纳入已有未提交内容：`workshop.ow`、`.zcode/`、`小丑牌地牢加无限时间.opy`、`海克斯乱斗.opy`。

## 优化原则

### 静态节流替代负载采样

将动态 `wait(...)` 表达式替换为其当前明确的静态下限或固定业务周期：

- `max(getAverageServerLoad() / 1000, X)` 改为 `wait(X)`，保留原有 `Wait` 行为参数。
- 启动路径中的 `wait(getAverageServerLoad() / 100)` 改为固定 `wait(0.25)`；已有以玩家槽位为基础的错峰保持不变，只移除负载采样加项。
- 移动规则从最低 0.016 秒动态循环统一为 `WAIT_SHORT_FRAME`（0.16 秒）。
- 英雄已有的 0.333、0.4、0.5、1 秒等业务节流保持不变，避免扩大玩法改动。

### 将昂贵查询限制在一次事件处理内

空间查询、LoS、玩家集合筛选和排序只能在已通过廉价门控后执行。单次规则或技能执行中，先构建一次候选列表，再复用该列表获取数量、首项或排序结果；禁止在条件和动作中对同一输入重复扫描。

### 限制特殊效果状态规模

持续效果采用“事件触发写入状态 -> 固定低频结算 -> 结束清空状态”的链路。动态目标、叠层、句柄与临时记录必须有容量上限，并在死亡、换英雄和 reset 路径清理。英雄 HUD 继续使用已有持久 `hudText` 槽位写入和清空，不恢复运行时创建/销毁方式。

## 分阶段路线图

### 阶段 1：退役运行时负载采样

**范围**：`modules/bootstrap`、`modules/ai`、受影响英雄规则、常规与 ARAM 模式设置。

**改动模式**：

- 删除所有有效及注释的 `getAverageServerLoad()` 调用；移除已禁用 Anti Crash 规则、相关设置初始化与编译期常量。
- 为避免 Workshop 公共变量索引漂移，保留 `global-vars.opy` 中既有 `ANTI_CRASH_*` 声明及其原有顺序，但不再读写这些槽位。
- 使用固定等待值替换动态等待；保留原有槽位错峰与 `Wait.ABORT_WHEN_FALSE`、`Wait.RESTART_WHEN_TRUE` 等控制语义。
- 同步修订性能规范和规则索引：项目策略改为禁止运行时服务器负载采样，使用静态节流、错峰和硬上限。

**风险与回滚**：固定等待可能使高负载场景下的响应速度略低，但不得改变技能效果。该阶段应单独提交；回滚只恢复采样相关提交，不与目标选择重构混合。

### 阶段 2：共享 AI 目标选择与移动节流

**范围**：`modules/ai/control/common.opy`、`modules/ai/movement/movement.opy` 及直接依赖其 `botTemp` / `botTarget` 的控制规则。

**改动模式**：

- 先以 Dummy、存活、已生成和英雄池等低成本条件门控；将半径、LoS 与候选筛选移至规则动作。
- 一次构建候选集合时同时完成存活、已生成、无敌、相位和特殊防御状态过滤；候选为空时清空 `botTarget`。
- 候选只有一人时直接使用；仅候选多于一人时按现有 `health × distance` 公式排序并取首项。
- 目标刷新固定为 0.512 秒；移动刷新固定为 `WAIT_SHORT_FRAME`。不跨 tick 缓存玩家对象，避免死亡、换英雄和位置变化产生陈旧目标。

**风险与回滚**：必须保持同一刷新窗口内的目标选择结果一致。若出现目标优先级、切换频率或寻路异常，仅回滚本阶段，不影响阶段 1。

### 阶段 3：按英雄簇消除重复玩家扫描

**范围与顺序**：

1. 支援与控制：Ana、Brigitte、Illari、Kiriko、Lucio、Mercy、Zarya。
2. 范围与持续效果：Mauga、Ramattra、Roadhog、Junker Queen、Juno、Hazard。
3. 高机动与瞄准：Sombra、Tracer、Pharah、Sojourn、Widowmaker，以及对应 AI control 文件。
4. ARAM 专有规则：只处理与常规赛同类的玩家扫描或动态等待；不改变 ARAM 覆盖玩法。

**改动模式**：

- 技能触发后只计算一次范围内/LoS 内候选数组；伤害量、治疗量、HUD 数字与目标选择复用同一结果。
- 将 `sorted(getLivingPlayers(...))` 改为“先筛选，后仅在多候选时排序”；保持既有排序键和目标阵营。
- 将大范围、字符串和多数组处理保留在事件触发或不低于既有业务周期的低频路径，不移入条件首项或无等待循环。
- 每次改动确认持续状态、生命池、HUD 与玩家引用在结束、死亡、换英雄和 reset 时回收。

**风险与回滚**：每个英雄簇独立提交；任何英雄出现目标不合法、效果残留、HUD 未清空或 ARAM 差异时，只回滚该英雄簇。

### 阶段 4：扫描工具与回归门禁

**范围**：`tools/perf-loop-scan.mjs`、相关测试/检查说明。

**改动模式**：

- 将有效 `getAverageServerLoad()` 调用列为严格扫描失败项，注释调用列为告警项。
- 保留既有无等待循环、条件顺序和热点报告；为同一规则块内重复空间查询或重复玩家集合构建增加可操作报告。
- 只报告可由源码可靠识别的模式；不把单次技能触发中的必要扫描误报为阻断问题。

**风险与回滚**：扫描器先以当前优化后源码为基线，避免把既有合法模式变为无关失败。若误报，缩小规则匹配而非放宽所有严格检查。

| 层级 | 检查 | 通过条件 |

## 提交与交付

- 阶段 1、阶段 2、每个英雄簇和阶段 4 分别使用小步、可回滚的 Conventional Commit。
- 每个提交说明记录受影响英雄/系统、预期负载变化，以及是否触及 init/reset 链路。
- 构建会改写受跟踪的 `workshop.ow`；除非提交明确包含生成产物，否则在交付前恢复该用户已有改动。
